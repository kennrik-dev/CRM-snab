# CRM Ultima — единое окно закупок · техническая спецификация

Документ описывает доменную модель, жизненный цикл заявки, вкладки, бизнес-правила и взаимодействия CRM по концепту `zakupki-crm-v5.html`. Предназначен для передачи ИИ-разработчику или команде. Целевой стек: **React (фронтенд) + FastAPI (бэкенд) + SQLite (БД)**.

Концепт-прототип — это фронтенд на ванильном HTML/CSS/JS с демонстрационными данными в памяти. Бэкенда, авторизации и реальной БД в нём нет — их нужно реализовать по этой спецификации.

---

## 1. Назначение

Единое окно, через которое отслеживается путь закупочной заявки: от создания комплектовщиком до полной оплаты и закрытия сделки. Заменяет общий Excel и разрозненную переписку. Ключевая идея — все отделы видят одни и те же заявки в одном месте, каждый со своими действиями и зоной ответственности.

**Отделы**: **Комплектация, Закупки, Сопровождение**. **Отдельные роли**: **Админ, Руководитель, Куратор**. Подробнее — раздел 1.1.

### 1.1. Пользователи, отделы и роли
Система рассчитана на небольшую команду — **до 20 пользователей**. Вход — по **логину и паролю, без самостоятельной регистрации**: учётные записи создаёт Админ.

**Доступ к редактированию данных — на уровне отдела.** В каждом отделе несколько сотрудников, и все они обладают одинаковыми правами своего отдела.

| Тип | Кто | Зона ответственности |
|---|---|---|
| Отдел | **Комплектация** | Создают заявки-родители, ведут позиции до передачи в закупку |
| Отдел | **Закупки** | Берут заявки в работу, торги на ЭТП, ветвление по поставщикам |
| Отдел | **Сопровождение** | Договоры, поставки, документы, УПД, **проведение оплаты и закрытие заявки** |
| Роль | **Админ** | Управление пользователями и системой *(детальные права — позже)* |
| Роль | **Руководитель** | Управленческий просмотр и отчёты *(детальные права — позже)* |
| Роль | **Куратор** | *(назначение и права — позже)* |

> Детальная матрица прав для ролей **Админ / Руководитель / Куратор** описывается отдельным шагом. Матрица действий по отделам в карточке заявки — в разделе 5.4.

> **Отличие от концепт-прототипа:** в прототипе (`zakupki-crm.js`) роль переключается выпадающим списком и присутствует отдельная роль «Оплата». В целевой системе вход — по логину/паролю, а действия бывшего отдела «Оплата» («Провести оплату», «Закрыть заявку») выполняет отдел **Сопровождение**.

### 1.2. Размещение
Хостинг пока не выбран — возможны два варианта: размещение **в интернете** (общий доступ) или **на локальном сервере** компании. Масштаб (до 20 пользователей) допускает оба варианта, и выбранный стек React + FastAPI + SQLite ему соответствует.

## 2. Доменная модель (сущности)

### 2.1. Заявка-родитель (Parent / «Т-67»)
Создаётся комплектовщиком. Имеет человекочитаемый код (`Т-67`).

| Поле | Тип | Описание |
|---|---|---|
| code | string | Код заявки, напр. `Т-67` (уникальный) |
| title | string | Наименование заявки |
| mtr | string | Тип МТР (категория) |
| srok | date | Желаемый срок поставки |
| zagruzka | date | Дата загрузки (создания комплектовщиком) |
| sostavitel | string | Составитель |
| dept | string | Отдел-комплектация (источник) |
| status | enum | `awaiting` (ожидает закупки) / `cancelled` (отмена) |
| positions[] | Position[] | Позиции заявки (пока не разбита) |
| children[] | Child[] | Дочерние заявки после ветвления |

### 2.2. Дочерняя заявка (Child / «1488», процедура «АП1488»)
Появляется при ветвлении на этапе закупки: **1 поставщик = 1 дочерняя заявка**. Дочерние ссылаются на родителя (`Т-67`), но дальше движутся по этапам **независимо**.

| Поле | Тип | Описание |
|---|---|---|
| num | string | Номер заявки, напр. `1488` |
| proc | string | Номер закупочной процедуры, напр. `АП1488` |
| parent_code | FK | Ссылка на родителя `Т-67` |
| supplier | string | Поставщик (или `—` пока не выбран) |
| lot | string | Лот |
| pubStart / pubEnd | date | Даты начала/завершения публикации на ЭТП |
| stage | enum | `proc` / `supp` / `late` / `pay` / `done` |
| contract | string | Номер и дата договора |
| statusZakup | string | Текстовый под-статус на этапе закупки (Приём заявок, Тех. оценка, Передать на сделку…) |
| statusSdelki | string | Статус сделки (Контракт заключён, На согласовании…) |
| statusPostavki | string | Статус поставки (Частично получена, Получена, Просрочена…) |
| srokDD | date | Срок поставки по договору (ДД) |
| plan / fakt | date | Плановая / фактическая дата |
| days | int | Дней на текущем этапе (для отчётов) |
| positions[] | Position[] | Позиции дочерней заявки |
| deliveries[] | Delivery[] | Поставки |

### 2.3. Позиция (Position / line item)
| Поле | Тип | Описание |
|---|---|---|
| name | string | Наименование |
| qty | number | Количество |
| unit | string | Ед. изм. (шт, м, кг, компл, м³…) |
| price | number | Цена за единицу |
| sum | computed | `qty * price` |
| delivery_ref | int / null | Номер поставки, в которую включена позиция. `null` = **ожидает отгрузки** (ещё не в поставке) |

### 2.4. Поставка (Delivery)
**Важное правило: поставка не может быть пустой.** Позиции без поставки лежат в блоке «ожидают отгрузки». Чтобы создать поставку, в неё добавляют позиции.

| Поле | Тип | Описание |
|---|---|---|
| n | int | Номер поставки внутри дочерней заявки (частичные поставки) |
| status | enum | `transit` (В поставке) / `done` (Получена) / `late` (Просрочена) |
| late | bool | Флаг «получена с задержкой» (для done) |
| date / eta | date | Факт получения / ожидаемая дата |
| positions[] | ref | Позиции, входящие в поставку |
| docs | object | `{ТТН, М-15, УПД, Серт}` — булевы признаки наличия документов |
| upd | string/null | Номер УПД (если выставлена) |
| payStatus | enum | `paid` / `await` / `late` / `none` |
| payDate | date | Дата оплаты |

### 2.5. УПД / платёж (Payment)
Выводится из поставки, у которой есть `upd`. Оплата отслеживается **по каждой УПД отдельно**.

Поля: `upd, supplier, parent_code, child_num, delivery_n, positions[], amount (Σ позиций поставки), payStatus, payDate, contract, docs`.

### 2.6. Комментарий (Comment)
Привязан к заявке (родительской по `code` или дочерней по `num`).

Поля: `request_ref, author, role, created_at, text`.

### 2.7. Отдел / роль пользователя
Отделы: `Комплектация | Закупки | Сопровождение`. Отдельные роли: `Админ | Руководитель | Куратор`. Отдел определяет доступные действия в карточке и подпись автора комментария. Подробнее — раздел 1.1.

## 3. Жизненный цикл заявки и ветвление

```
[Создана]            Комплектация создаёт Т-67 (позиции, срок). Статус: Ожидают закупки.
   │                 (возможна Отмена)
   ▼
[В закупке]          Закупки берут в работу → размещают на ЭТП → отбор поставщиков,
   │                 тех. оценка, уторговывание.
   │  ⎇ ВЕТВЛЕНИЕ:   по итогам торгов Т-67 разбивается на дочерние заявки по поставщикам:
   │                 1488/1489/1490 (процедуры АП1488/АП1489/АП1490). 1 поставщик = 1 заявка.
   ▼
[В сопровождении]    По каждой дочерней: заключается договор, контролируются ПОСТАВКИ
   │                 (частичные, у каждой свой статус), приём документов (ТТН, М-15, УПД, Серт),
   │                 подписание УПД.
   ▼
[В оплате]           Когда поставки получены и документы собраны — оплата по каждой УПД.
   ▼
[Закрыта]            Все позиции доставлены, все УПД оплачены.
```

**Ключевые следствия ветвления для UI:**
- После разбивки `Т-67` исчезает из единого списка как одна строка и появляется как 3 отдельные дочерние заявки (1488/1489/1490), **каждая с пометкой родителя `Т-67`**.
- Дочерние заявки одного родителя могут одновременно находиться на **разных этапах** (напр. 1488 — в сопровождении, 1489 — в оплате, 1490 — ещё в закупке).
- В карточке дочерней заявки есть переключатель между «сёстрами» (1488 ⇄ 1489 ⇄ 1490).

## 4. Вкладки (страницы) и их назначение

| Вкладка | Назначение | Содержимое |
|---|---|---|
| **Дашборд** | Сжатая сводка | 6 показателей (сегментные индикаторы), поток по этапам, «Требует внимания», «Лента событий», **компактные** таблицы по этапам |
| **Комплектация** | Заявки от комплектовщиков, ожидающие закупки | Полная таблица (см. 4.1) |
| **В закупке** | Заявки, взятые закупщиками в работу (ЭТП) | Полная таблица (см. 4.2) |
| **В сопровождении** | Заявки после завершения закупки — контроль поставок и документов | Полная таблица (см. 4.3) |
| **Оплаты** | Финансы и реестр платежей | Сводка + реестр УПД + карточка платежа |
| **Отчёты** | Конструктор выгрузок для руководства | 4 типа отчётов + параметры + экспорт |
| **Карточка заявки** | Открывается из любой строки | Детали заявки (см. 4.6) |

### 4.1. Комплектация (= «Ожидают закупки»)
Колонки: `#, Наименование заявки (Т-67 + название), Тип МТР, Срок поставки, Дата загрузки, Составитель, Позиции, Статус`.
Статус по умолчанию — «Ожидают закупки», возможен «Отмена».

### 4.2. В закупке
Колонки: `#, Наименование заявки (Т-67), Тип МТР, № заявки, № процедуры, Поставщик, Дата загрузки, Дата начала публикации, Дата завершения публикации, Поз., Статус`.

### 4.3. В сопровождении
Колонки: `#, Наименование заявки (Т-67), № заявки, № процед., Поставщик, Тип МТР, Сумма договора, Статус сделки, Статус поставки, Срок ДД, План, Факт, Просроч., Документы, Поз.`
Таблица должна умещаться **без горизонтального скролла** (фикс. раскладка, перенос длинных значений).

### 4.4. Дашборд — компактные таблицы (отличаются от страниц!)
Дашборд = краткий обзор, поэтому колонок меньше:
- Ожидают закупки: `#, Наименование, Тип МТР, Срок, Поз., Статус`
- В закупке: `#, Наименование, № заявки, Поставщик, Поз., Статус`
- В сопровождении: `#, Наименование, № заявки, Поставщик, Сумма договора, Статус поставки, Просроч., Прогресс`

### 4.5. Оплаты
- Сводка: «Сумма в работе», «Оплачено», «К оплате», «Просрочено» + полоса распределения (оплачено / выставлено-ждёт / поставлено-без-УПД / законтрактовано-не-поставлено).
- Реестр платежей: `УПД, Заявка (Т-67/1488), Поставщик, Поставка, Статус, Срок, Сумма`.
- Карточка платежа: УПД, договор, поставщик, связанная поставка, документы, **позиции в составе УПД**, действие «Провести оплату» (только отдел Сопровождение).

### 4.6. Карточка заявки
- Шапка: код родителя `Т-67` + переключатель сестёр; номер дочерней `1488` + процедура `АП1488`; поставщик, лот, договор, тип МТР; сумма договора.
- Маршрут заявки (5 шагов): Создана → В закупке → Сопровождение → Оплата → Закрыта.
- Прогресс позиций: «Поставлено X из Y» (заявка не закрыта, пока не доставлены все позиции).
- Панель действий, зависящая от **роли** (см. 5.4).
- **Поставки и позиции**: каждая поставка — таблица позиций (Наименование, Кол-во, Ед., Цена, Сумма) + документы (ТТН, М-15, УПД, Серт) + статус оплаты.
- **Ожидают отгрузки**: позиции без поставки (кнопка «Создать поставку»).
- **Комментарии**: лента + поле ввода (автор = текущий отдел).

### 4.7. Отчёты
Параметры: тип отчёта, период, фильтры (тип МТР, поставщик, составитель). Экспорт: Excel / PDF / CSV.
Типы:
1. **Время на этапе / зависания** — список заявок с текущим этапом и числом дней на нём; флаг «зависло > N дней».
2. **Суммы по этапам и поставщикам** — агрегаты сумм договоров.
3. **Просрочки** — просроченные поставки и оплаты.
4. **Сводка по составителям/отделам** — кол-во заявок и суммы по людям.

## 5. Бизнес-правила

### 5.1. Прогресс и закрытие
Прогресс = доставленные позиции / всего позиций дочерней заявки. Заявка не закрывается, пока не доставлены все позиции и не оплачены все УПД.

### 5.2. «Просроч., %» (колонка в «В сопровождении»)
Доля позиций с нарушением срока = (позиции в поставках со статусом «Просрочена» **или** «получена с задержкой») / всего позиций × 100.
Пример: 10 позиций, 2 в просроченной поставке → 20%. Цвет: 0% — зелёный, >0 — оранжевый, ≥50% — красный.

### 5.3. «Документы»
Документ считается полученным, если он есть **во всех** поставках заявки. Набор: ТТН, М-15, УПД, Серт. В таблице — 4 квадрата (Т/М/У/С), зелёный = есть, красный = нет.

### 5.4. Права по отделам (действия в карточке)
| Этап \ Отдел | Комплектация | Закупки | Сопровождение |
|---|---|---|---|
| Ожидают закупки | Редактировать позиции, Дублировать, Отменить | Взять в работу | — |
| В закупке | — | На ЭТП, Разбить по поставщикам, Выбрать поставщика, Передать в сопровождение | — |
| В сопровождении | — | — | Создать поставку, Отметить документы, Подписать УПД, Передать в оплату |
| В оплате | — | — | Провести оплату, Закрыть заявку |

Права ролей **Админ / Руководитель / Куратор** — отдельным шагом (см. 1.1).

### 5.5. Оплата
По каждой УПД отдельно. Остаток к оплате = Выставлено (Σ УПД) − Оплачено. Просрочка оплаты отслеживается отдельно.

### 5.6. Поставка не пустая
Создание поставки = добавление в неё ≥1 позиции из «ожидают отгрузки». Поставка получает статус В поставке / Получена / Просрочена.

## 6. Статусы и цветовая семантика
| Статус | Цвет |
|---|---|
| Ожидают закупки / Создана | серый |
| В закупке | оранжевый |
| В сопровождении / В поставке | синий |
| В оплате | фиолетовый |
| Получена / Оплачена / Закрыта / в срок | зелёный |
| Просрочено (поставка/оплата) | красный |
| Отмена | серый, зачёркнуто |

## 7. Рекомендации по реализации

### 7.1. Схема БД (SQLite, упрощённо)
```sql
CREATE TABLE parent_request (
  id INTEGER PRIMARY KEY, code TEXT UNIQUE, title TEXT, mtr TEXT,
  srok DATE, zagruzka DATE, sostavitel TEXT, dept TEXT,
  status TEXT CHECK(status IN ('awaiting','cancelled')) DEFAULT 'awaiting'
);
CREATE TABLE child_request (
  id INTEGER PRIMARY KEY, num TEXT UNIQUE, proc TEXT,
  parent_id INTEGER REFERENCES parent_request(id),
  supplier TEXT, lot TEXT, pub_start DATE, pub_end DATE,
  stage TEXT CHECK(stage IN ('proc','supp','late','pay','done')),
  contract TEXT, status_sdelki TEXT, status_postavki TEXT,
  srok_dd DATE, plan_date DATE, fakt_date DATE, stage_entered_at DATE
);
CREATE TABLE position (
  id INTEGER PRIMARY KEY, owner_kind TEXT, owner_id INTEGER,
  name TEXT, qty REAL, unit TEXT, price REAL, delivery_id INTEGER NULL
);
CREATE TABLE delivery (
  id INTEGER PRIMARY KEY, child_id INTEGER REFERENCES child_request(id),
  n INTEGER, status TEXT, late INTEGER DEFAULT 0, date DATE, eta DATE,
  doc_ttn INTEGER, doc_m15 INTEGER, doc_upd INTEGER, doc_sert INTEGER,
  upd TEXT, pay_status TEXT, pay_date DATE
);
CREATE TABLE comment (
  id INTEGER PRIMARY KEY, request_kind TEXT, request_id INTEGER,
  author TEXT, role TEXT, text TEXT, created_at DATETIME
);
```
> Появилась авторизация (раздел 1.1), поэтому потребуется таблица **`user`** (логин, хэш пароля, ФИО, отдел/роль). Её точная структура и привязка прав описываются на шаге про права ролей.

### 7.2. Эндпоинты FastAPI (черновик)
```
GET  /requests?stage=...            — список (с фильтром по этапу)
GET  /requests/{code}               — родитель + дочерние
GET  /children/{num}                — дочерняя со всеми поставками/позициями
POST /children/{num}/split          — ветвление по поставщикам
POST /children/{num}/deliveries     — создать поставку (добавить позиции)
PATCH /deliveries/{id}              — статусы, документы, УПД
GET  /payments                      — реестр платежей (УПД)
POST /payments/{upd}/pay            — провести оплату
GET  /reports/{type}                — отчёты (time|sums|late|people)
GET/POST /requests/{ref}/comments   — комментарии
```

### 7.3. Структура React-компонентов
```
<App> (роль, роутер вкладок)
 ├─ Dashboard (Meters, FlowRail, AttentionPanel, FeedPanel, CompactTables)
 ├─ StagePage[kompl|zakup|soprov] (StageTable со своими колонками)
 ├─ PaymentsPage (BalanceSummary, PaymentsRegister) → PaymentCard
 ├─ ReportsPage (ParamsPanel, ReportOutput)
 └─ RequestCard (Header+Siblings, Route, Progress, RoleActions,
                 DeliveryBlock[Positions, Docs], AwaitingBlock, Comments)
```
Состояние ролей и фильтров — в контексте; данные — через API. Статусы/цвета вынести в общие токены (см. `:root` в `zakupki-crm.css`).
