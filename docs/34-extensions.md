# 34 · Точки расширения (планируемые интеграции)

Возможности, **заложенные в архитектуру, но реализуемые позже**. Здесь — что именно закладываем сейчас, чтобы последующее подключение не потребовало переделки модели данных. В MVP эти разделы **не реализуются**; задача документа — гарантировать, что дизайн их не блокирует.

Принцип: чем дешевле «дозаложить» позже, тем меньше делаем сейчас. В SQLite добавить столбец (`ALTER TABLE ADD COLUMN`) и новую таблицу — дёшево; изменить `CHECK`-ограничение — дорого (нужна пересборка таблицы). Поэтому **сейчас** трогаем только то, что иначе потребует пересборки.

---

## 1. Внешний учёт оплат (интеграция с 1С и т.п.)

**Цель:** статус оплаты УПД обновляется автоматически из внешней системы (1С), а не только вручную сотрудником Сопровождения.

**Заложено сейчас (`30-db-schema.md`):**
- `upd_payment.origin` допускает значение `'external'` (помимо `'delivery'` / `'manual'`) — заранее, т.к. менять `CHECK` потом дорого.
- `upd_payment.ext_source` — какая система (напр. `'1c'`); `upd_payment.ext_id` — идентификатор документа во внешней системе (для сопоставления и идемпотентности). Пока всегда `NULL`.
- Смена `pay_status` инкапсулирована в **одну сервисную операцию** (та же, что вызывает кнопка «Провести оплату»). Внешний источник вызывает её же — UI и интеграция идут одним путём.

**Реализуем позже:**
- Входящий эндпоинт `POST /integrations/1c/payments` **или** периодический опрос 1С — сопоставление по `ext_id` / № УПД, проставление `paid` + `pay_date`.
- Индекс `ix_upd_ext ON upd_payment(ext_source, ext_id)` для сопоставления (добавляется при реализации).
- Аутентификация интеграции — **сервисный токен**, а не пользовательская cookie.
- Журналирование в `audit_log` (исполнитель = «система / 1С»).
- **Уточнить при реализации:** направление обмена — webhook из 1С → CRM или опрос 1С из CRM (зависит от возможностей конкретной 1С).

## 2. Документы (вложения) к заявке

**Цель:** прикреплять файлы (сканы, спецификации, договоры) к заявке / процедуре / поставке / УПД.

**Заложено сейчас:** ничего менять не нужно — используется та же **полиморфная привязка** (`entity_kind` + `entity_id`), что уже есть у `comment` и `audit_log`. Файлы хранятся **на диске**, в БД — только метаданные и путь (SQLite не предназначен для блобов).

**Реализуем позже (готовая DDL):**
```sql
CREATE TABLE attachment (
  id INTEGER PRIMARY KEY,
  entity_kind TEXT NOT NULL,        -- parent | procedure | delivery | upd_payment
  entity_id   INTEGER NOT NULL,
  filename    TEXT NOT NULL,        -- исходное имя файла
  path        TEXT NOT NULL,        -- путь в файловом хранилище
  mime        TEXT,
  size        INTEGER,              -- байт
  uploaded_by INTEGER REFERENCES user(id),
  created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX ix_attach_entity ON attachment(entity_kind, entity_id);
```
- Эндпоинты: `POST /{entity}/{id}/attachments` (multipart), `GET …/attachments`, `GET /attachments/{id}` (скачать), `DELETE /attachments/{id}`.
- В карточках (`16` / `17`) — блок «Документы (файлы)» рядом с «Историей».
- Права — как на редактирование соответствующего блока; удаление — автор/Куратор/Админ.
- **Бэкап** (`33-nonfunctional.md` §3): включить каталог файлов в резервную копию вместе с БД.

## 3. Уведомления (почта / мессенджер)

**Цель:** оповещать пользователей о событиях — заявка взята в работу, передача в сопровождение, просрочка поставки/оплаты и т.п. — по email или мессенджеру.

**Заложено сейчас:**
- События уже перечислены в `audit_log` (`33` §2) и в панели «Требует внимания» (`32-calculations.md`) — уведомления подписываются на их **подмножество**, новый учёт событий не нужен.
- У пользователя есть `email` (логин) как канал по умолчанию. Контакт мессенджера и настройки добавляются при реализации дешёвым `ALTER TABLE ADD COLUMN` (напр. `user.messenger_id`).

**Реализуем позже:**
- Каналы: email (SMTP) и/или мессенджер (напр. Telegram-бот) — выбрать при реализации.
- Таблица-настройки: какие события → каким ролям/пользователям, канал, вкл/выкл.
- **Outbox** (отдельная таблица + фоновый воркер) для надёжной доставки и ретраев.
- Конфиг (SMTP-доступ / токен бота) — в параметрах развёртывания, не в коде.
- Триггеры: запись определённых типов в `audit_log` + проверка производных условий (просрочка) по расписанию.
