# CRM Ultima — документация для реализации

Рабочая документация проекта. Цель — описать систему настолько точно, чтобы её можно было реализовать без устных уточнений. Целевой стек: **React (фронтенд) + FastAPI (бэкенд) + SQLite (БД)**.

- Высокоуровневый обзор: [`../Concept design/SPECIFICATION.md`](../Concept%20design/SPECIFICATION.md)
- Концепт-прототип: [`../Concept design/index.html`](../Concept%20design/index.html) (`zakupki-crm.css`, `zakupki-crm.js`)

---

## Принятые решения (Этап 0 ✅)

- **Название:** CRM Ultima.
- **Отделы** (доступ к данным — на уровне отдела, в каждом несколько сотрудников): Комплектация, Закупки, Сопровождение.
- **Отдельные роли:** Админ, Руководитель, Куратор (детальные права — Этап 1.3).
- **Вход:** логин + пароль, без самостоятельной регистрации (учётные записи создаёт Админ).
- **Оплата:** действия «Провести оплату» и «Закрыть заявку» выполняет отдел **Сопровождение** (отдельного отдела «Оплата» нет).
- **Размещение:** интернет или локальный сервер (не решено). Масштаб — до 20 пользователей.
- **Заложены на будущее** (реализация позже): внешний учёт оплат (1С), вложения-файлы к заявкам, уведомления по почте/мессенджеру — см. [`34-extensions.md`](34-extensions.md).
- **Запуск и эксплуатация:** первый Админ предзаполнен (дефолтный пароль, смена при первом входе); часовой пояс — **Москва**; конкурентность — last-write-wins; списки — активные по умолчанию (архив под переключателем), пагинация 50, глобальный поиск в шапке; справочники статусов — с начальным seed (НДС-разбивка — при интеграции с 1С).

## Дорожная карта

Принцип: идём от общего фундамента к деталям — страницы ссылаются на одни и те же сущности, статусы и роли, поэтому их фиксируем первыми.

| Этап | Содержание | Критерий готовности | Статус |
|---|---|---|---|
| 0 | Сверка видения, базовые решения | Решения зафиксированы (см. выше) | ✅ |
| 1 | **Сквозные основы:** доменная модель, статусы и переходы, права доступа, авторизация | Каждая сущность/статус/роль определены однозначно | ✅ |
| 2 | **Постраничная детализация** по единому шаблону | Колонки и действия совпадают с концептом; действия связаны со статусами и ролями | ✅ |
| 3 | **Технический контракт:** схема БД, API, расчётные правила, нефункциональное | Каждое UI-поле имеет источник в схеме и в API | ✅ |
| 4 | **Сборка и приёмка:** вычитка, сверка с прототипом, чек-лист готовности | Нет ссылок на несуществующие статус/роль/поле | ✅ |

## Шаблон описания страницы (Этап 2)

Каждая страница документируется по одному шаблону:

1. **Назначение и доступ** — что за страница, кто её видит.
2. **Макет/зоны экрана** — по концепту.
3. **Таблица** — колонки (источник данных, формат), сортировка, фильтры, поиск.
4. **Действия** — кнопки → кто может → что происходит → переход статуса.
5. **Навигация** — куда ведут клики.
6. **Состояния** — пусто / загрузка / ошибка / нет прав.
7. **Бизнес-правила и валидация**, специфичные для страницы.
8. **Открытые вопросы.**
