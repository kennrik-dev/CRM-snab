# 01 · Доменная модель

Финальная доменная модель CRM Ultima: сущности, связи, ключевые поля и инварианты. Концептуальный уровень (физическая схема БД — в `30-db-schema.md`).

> **Уточняет и заменяет** разделы 2–3 `../Concept design/SPECIFICATION.md`: там модель **двухуровневая** (родитель → дочерняя, «1 поставщик = 1 дочерняя»). Здесь — **трёхуровневая** (родитель → торг → процедура). Колонки страниц и §2–3 спецификации будут приведены в соответствие на шаге работы со страницами / при финальной сборке.

---

## 1. Иерархия (3 уровня)

```
Заявка-родитель   Т-67          — комплектовщик; полный список запрошенных позиций
   └─1:N─ Торг (№ заявки)  №1488 — закупщик; на одном Т-67 их может быть несколько
            └─1:N─ Процедура (№ процедуры)  №АП115116 — закупщик; на одном торге их может быть несколько
                     = 1 поставщик + 1 договор + свои позиции + поставки + УПД
```

- **Заявка-родитель** заводится комплектовщиком, имеет главный код `Т-67`.
- **Торг** создаёт закупщик, разбивая родителя на одну или несколько закупок (№ заявки). Группирует подмножество позиций родителя, выносимых на ЭТП вместе.
- **Процедура** — рабочая единица, отслеживаемая независимо (бывшая «дочерняя заявка»). По итогам торгов закупщик создаёт по процедуре на каждого поставщика. **1 процедура = 1 поставщик = 1 договор.**

Пример: торг `№1488` содержит кабели, моторы, шкафы. После ЭТП:
`№1488 / АП115116 / Поставщик №1` — кабели · `№1488 / АП115117 / Поставщик №2` — моторы · `№1488 / АП115118 / Поставщик №3` — шкафы.

## 2. Сущности

### 2.1. Заявка-родитель (ParentRequest) — `Т-67`
Создаёт **Комплектация**.

| Поле | Тип | Описание |
|---|---|---|
| code | string, уникальный | Главный код заявки `Т-67`, вводится вручную комплектовщиком |
| title | string | Наименование заявки |
| mtr | string (текст) | Тип МТР |
| srok | date | Желаемый срок поставки |
| zagruzka | date | Дата загрузки (создания) |
| sostavitel | string | ФИО составителя |
| dept | string | Отдел-источник (Комплектация) |
| status | enum | `awaiting` (Ожидает закупки) / `cancelled` (Отменена — остаётся в Комплектации). См. `02-statuses.md` §2 |

Связи: содержит **запрошенные позиции** (до распределения по торгам); имеет 1..N **торгов**.

### 2.2. Торг (Tender) — `№ заявки`, напр. `№1488`
Создаёт **Закупки**.

| Поле | Тип | Описание |
|---|---|---|
| num | string, уникальный | № заявки `№1488`, вводится вручную закупщиком |
| parent_ref | FK → ParentRequest | Ссылка на родителя `Т-67` |

Связи: ссылается на родителя; группирует подмножество запрошенных позиций; **всегда содержит ≥1 процедуру**. Без дробления — одна **процедура по умолчанию**, которую интерфейс показывает как саму заявку; при дроблении (разная номенклатура / несколько поставщиков) добавляются ещё процедуры.
Собственного этапа у торга **нет** — его состояние выводится из процедур.

### 2.3. Процедура (Procedure) — `№ процедуры`, напр. `№АП115116`
Создаёт **Закупки**. Рабочая единица, движется по этапам независимо от «сестёр».

| Поле | Тип | Описание |
|---|---|---|
| proc | string, уникальный | № процедуры `№АП115116`, вводится вручную закупщиком |
| tender_ref | FK → Tender | Ссылка на торг `№1488` |
| supplier | string (текст) | Поставщик (1 процедура = 1 поставщик); `—` пока не определён |
| fio_zakupshchik | string | ФИО закупщика — вводится в карточке «В закупке» |
| mtr | string (текст) | Тип МТР — наследуется от родителя, **закупщик может переопределить вручную** при дроблении |
| pub_start / pub_end | date | Даты начала/завершения публикации на ЭТП |
| contract | string | Номер и дата договора |
| fio_dogovornik | string | ФИО договорника — вводится в карточке «В сопровождении» |
| contract_sum | money (₽) | Сумма договора — авто Σ позиций, **правится вручную** |
| block | enum | Текущий блок: `zakupka` (В закупке) / `soprovozhdenie` (В сопровождении) |
| status_zakup | enum (справочник) | Статус в «В закупке»: Новая…На сделку / Отменена. Выставляет Закупки. См. `02-statuses.md` §3 |
| status_postavki | enum (справочник) | Статус поставки в «В сопровождении» — **вручную**: Новая…Поставлено / Отменена. См. `02-statuses.md` §4.1 |
| status_sdelki | enum (справочник) | Статус сделки (информативный): Согласование / Подготовка ДД / Подписано. См. `02-statuses.md` §4.2 |
| srok_dd | date | Срок поставки по договору (ДД) |
| plan / fakt | date | Плановая / фактическая дата |

Связи: ссылается на торг; содержит **позиции процедуры** и **поставки**.
Признак «просрочена» — **производный**, не этап (см. §3).

### 2.4. Позиция (Position)
Существует в двух ролях:

**(а) Запрошенная позиция** — на уровне родителя, «что нужно» (заводит комплектовщик).

| Поле | Тип | Описание |
|---|---|---|
| name | string | Наименование |
| qty | number | Запрошенное количество |
| unit | string (текст) | Ед. изм. (шт, м, кг, компл, м³…) |
| gost_tu | string | ГОСТ / ТУ |
| doc_code | string | Шифр документации |

**(б) Позиция процедуры** — на уровне процедуры, «что закупается у поставщика».

| Поле | Тип | Описание |
|---|---|---|
| source_ref | FK → запрошенная позиция / null | Из какой запрошенной позиции взята (наследует ГОСТ/ТУ и шифр документации). `null` — позицию добавил закупщик заново |
| name | string | Наименование |
| qty | number | Количество в этой процедуре (**может быть частью** запрошенного — дробление) |
| unit | string (текст) | Ед. изм. |
| price | money (₽, **с НДС**) | Цена за единицу |
| sum | computed | `qty * price` |
| delivery_ref | FK → Delivery / null | В какую поставку включена. `null` = **ожидает отгрузки** |

**Инвариант дробления:** для одной запрошенной позиции Σ(qty позиций процедур, ссылающихся на неё) ≤ qty запрошенной. Количество одной позиции может уходить разным поставщикам (разным процедурам) или одному целиком — решает закупщик по итогам ЭТП.

**Ведение позиций (через карточку):** **добавить / изменить / удалить / копировать** строку + массовая **вставка из Excel**. Комплектовщик ведёт запрошенные позиции (карточка «Комплектация»); закупщик ведёт позиции процедуры и проставляет **цену (с НДС)** до передачи в сопровождение. Добавленные «с нуля» позиции имеют `source_ref = null`.

### 2.5. Поставка (Delivery) — на уровне процедуры
**Поставка не может быть пустой** — создаётся добавлением ≥1 позиции из «ожидают отгрузки».

| Поле | Тип | Описание |
|---|---|---|
| n | int | № поставки внутри процедуры (частичные поставки) |
| status | enum | `transit` (в поставке) / `done` (получена) |
| late | bool (производный) | Получена с задержкой (см. §3) |
| date / eta | date | Факт получения / ожидаемая дата |
| docs | {ttn, m15, upd, sert}: bool | Наличие документов: ТТН, М-15, УПД, Серт |
| upd | string / null | № УПД — вводит вручную **Сопровождение**, когда поставщик выставил УПД |
| pay_status | enum | `await` (Ожидает оплаты) / `paid` (Оплачено); `none` пока УПД не выставлена. «Просрочена» — производная. См. `02-statuses.md` §5 |
| pay_date | date | Дата оплаты |

> Поля `upd` / `pay_status` / `pay_date` показываются на поставке, но **физически хранятся в `upd_payment`** (связь `delivery_id`), а не в таблице поставки — в карточке они «подтягиваются» из платежа. Схема — `30-db-schema.md`.

Связи: принадлежит процедуре; включает позиции процедуры.

### 2.6. УПД / Платёж (Payment)
Оплата отслеживается **по каждой УПД отдельно**. УПД появляется **двумя путями**:
- **из поставки** — Сопровождение ввело № УПД в карточке (тянет поставщика/договор/позиции из процедуры);
- **вручную в «Оплаты»** — добавлена прямо в реестр платежей (поля вводятся руками).

| Поле | Тип | Описание |
|---|---|---|
| upd | string | № УПД |
| origin | enum | `delivery` (из поставки) / `manual` (добавлена вручную) |
| request_label | string | Заявка — `Т-67 + №…` (ссылка на процедуру для `delivery`; свободный текст для `manual`) |
| supplier | string | Поставщик (из процедуры или ручной ввод) |
| contract | string | Договор — тянется из «В сопровождении» или ручной ввод (нет данных — прочерк) |
| zrds | string | № ЗРДС — ручной ввод (нет данных — прочерк) |
| srok | date | Срок оплаты — **ручной ввод** |
| amount | money (₽) | Сумма с НДС — авто Σ позиций (для `delivery`) или ручной ввод (для `manual`) |
| positions[] | line[] | Позиции в составе УПД. Для `manual` — опционально: №, наименование, ед. изм., кол-во, цена за ед., сумма |
| pay_status / pay_date | enum / date | `await` (Ожидает оплаты) / `paid` (Оплачено) + производная «Просрочена». См. `02-statuses.md` §5 |
| delivery_ref | FK / null | Поставка-источник (для `delivery`) |
| docs | {ttn,m15,upd,sert} | Документы поставки (для `delivery`) |

### 2.7. Комментарий (Comment)
Привязан к любому уровню — родителю (`code`), торгу (`num`) или процедуре (`proc`).

Поля: `target_ref, target_kind, author (ФИО), role (отдел), created_at, text`.

### 2.8. Пользователь (User)
Кратко (права — `03-roles-permissions.md`, вход — `04-auth.md`): `login, password_hash, ФИО`. Пользователь — либо **член отдела** (Комплектация / Закупки / Сопровождение) как **обычный сотрудник** или **Куратор**, либо **глобальная роль** (Админ / Руководитель).

## 3. Инварианты и доменные правила

1. **Кардинальности:** Родитель `1:N` Торг `1:N` Процедура (Торг всегда содержит **≥1** процедуру; без дробления — одна процедура по умолчанию, отображается как сама заявка). Поставка и позиции процедуры — на уровне процедуры.
2. **1 процедура = 1 поставщик = 1 договор.**
3. **Дробление позиций:** количество запрошенной позиции распределяется по процедурам (целиком или частями); Σ распределённого ≤ запрошенного.
4. **Деньги:** валюта всегда ₽; цена позиции — **с НДС**; сумма договора и сумма УПД считаются автоматически (Σ позиций), но сотрудник может править вручную; в денежные поля вводятся только цифры.
5. **Просрочка — производный признак, не этап.** Появляется, когда срок по ДД раньше факта (`srok_dd < fakt`) для процедуры или когда позиция/поставка получена позже срока. Влияет на флаг `late` поставки и колонку «Просроч.».
6. **Поставка не пустая:** создание = добавление ≥1 позиции из «ожидают отгрузки».
7. **Прогресс и завершённость — на уровне процедуры:** прогресс = доставленные позиции / всего позиций процедуры; процедура **завершена**, когда статус поставки = `Поставлено` и все её УПД = `Оплачено` (отдельного статуса «Закрыта» нет, см. `02-statuses.md` §5). Родитель `Т-67` завершён, когда завершены все его процедуры.
8. **Идентификаторы вводятся вручную:** `Т-67` — комплектовщик; `№ заявки` и `№ процедуры` — закупщик; `№ УПД` — сопровождение.

## 4. Открытые вопросы и решения

- ✅ **Цена позиции:** комплектовщик вводит запрошенные позиции (наименование, кол-во, ед. изм.) **без цены**; цену (с НДС) проставляет закупщик на уровне позиции процедуры по итогам торгов.
- ✅ **Тип МТР:** задаётся на уровне родителя (обязательно); при дроблении закупщик может **переопределить** МТР на каждой процедуре вручную. Дата загрузки и желаемый срок поставки — только на уровне родителя.
- ✅ **Лот:** убран. В трёхуровневой модели роль «лота» полностью покрывают Торг + Процедура + Поставщик; отдельное поле не заводим.
- ✅ **Без дробления:** даже когда вся номенклатура у одного поставщика, внутри всегда одна **процедура по умолчанию** (поставщик/договор/поставки всегда на процедуре, единообразно); интерфейс показывает её как саму заявку. Дробление = добавление процедур.
