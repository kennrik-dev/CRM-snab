# 02 · Статусы и переходы (блочная модель)

Статусы организованы **по блокам** (страницам). Каждый блок имеет свой набор статусов.

**Где живут статусы:**
- В блоке **«Комплектация»** статус несёт **заявка-родитель** `Т-67`.
- В блоках **«В закупке»** и **«В сопровождении»** статус несёт **процедура**. После разбивки `Т-67` на процедуры у каждой свой статус, и они движутся **независимо**. Без дробления у заявки одна **процедура по умолчанию** (интерфейс показывает её как саму заявку); при дроблении появляются дополнительные процедуры, каждая со своим статусом.
- **«Оплаты»** — **параллельный реестр УПД**: единица здесь — УПД, а не процедура.

См. сущности в `01-domain-model.md`, права — в `03-roles-permissions.md`.

---

## 1. Карта блоков

| Блок | Единица | Основной статус | Доп. статус |
|---|---|---|---|
| Комплектация | Заявка-родитель | Ожидает закупки · Отменена | — |
| В закупке | Процедура | Новая → … → На сделку · Отменена | — |
| В сопровождении | Процедура | **Статус поставки**: Новая → … → Поставлено · Отменена | **Статус сделки** (информативный) |
| Оплаты | УПД | Ожидает оплаты · Оплачено | Просрочена (производный) |

---

## 2. Комплектация — статус заявки-родителя

| Статус | Значение |
|---|---|
| `Ожидает закупки` | Создана комплектовщиком, ждёт, когда закупщик возьмёт в работу (начальный) |
| `Отменена` | Заявка отменена. **Остаётся в блоке «Комплектация»** (для отчётности/статистики), дальше не движется |

**Переход из блока:** «Взять в работу» (Закупки) → заявка уходит в «В закупке» со статусом `Новая`.
Отмену ставит **Комплектация** (заявка `Ожидает закупки` → `Отменена`).

---

## 3. В закупке — статус процедуры

Закупщик разбивает заявку на торги (№ заявки) и процедуры (№ процедуры, по поставщикам); после разбивки каждая процедура движется по статусам независимо.

**Рабочая последовательность** (справочник, выставляет Закупки):
```
Новая → Приём заявок → Торги → Тех. экспертиза → Дозапросы → Согласование → На сделку
```
+ особый статус `Отменена`.

| Статус | Примечание |
|---|---|
| `Новая` | Начальный — процедура только зашла в блок |
| `Приём заявок` · `Торги` · `Тех. экспертиза` · `Дозапросы` · `Согласование` | Рабочие статусы хода закупки на ЭТП |
| `На сделку` | **Финальный в блоке.** Закупщик может «Передать в сопровождение» |
| `Отменена` | Процедура **остаётся в блоке «В закупке»**; закупщик может **вернуть любой рабочий статус** и продолжить работу (обратимо) |

**Переход из блока:** «Передать в сопровождение» (Закупки), из статуса `На сделку` → процедура уходит в «В сопровождении» со статусом `Новая`.

> **Служебные значения:** `Новая` (ставит система при входе в блок) и `Отменена` (ставит действие «Отменить процедуру») — **не входят в справочник** и не выбираются из списка вручную. Закупщик выбирает только рабочие статусы между ними (`Приём заявок … На сделку`). Начальный seed справочника — `30-db-schema.md`.

---

## 4. В сопровождении — два статуса процедуры

### 4.1. Статус поставки (основной, **вручную**)
Выставляет Сопровождение, выбором из списка (`В производстве`/`В поставке` нельзя вычислить автоматически):
```
Новая → В производстве → В поставке → Частично поставлено → Поставлено
```
+ особый статус `Отменена`.

| Статус | Значение |
|---|---|
| `Новая` | Процедура только пришла из закупки (начальный) |
| `В производстве` | Поставщик изготавливает |
| `В поставке` | Отгружено / в пути |
| `Частично поставлено` | Получена часть позиций |
| `Поставлено` | Получены все позиции |
| `Отменена` | Особый статус (поведение — см. §8, открытый вопрос) |

### 4.2. Статус сделки (побочный, информативный)
Не управляет переходами, показывает оформление договора:
```
Согласование → Подготовка ДД → Подписано
```

---

## 5. Оплаты — параллельный реестр УПД

Единица — **УПД**. УПД появляется в реестре, как только Сопровождение её выставило (ввело № УПД). Процедура при этом **остаётся в «В сопровождении»** (частичные поставки и УПД).

| Статус | Значение |
|---|---|
| `Ожидает оплаты` | УПД выставлена, не оплачена (начальный) |
| `Оплачено` | Оплата проведена |
| `Просрочена` | **Производный доп. признак** поверх `Ожидает оплаты`: срок оплаты прошёл, а статус ещё не `Оплачено` |

**Переход:** «Провести оплату» (Сопровождение) → `Ожидает оплаты` → `Оплачено`.

> Отдельного статуса «Закрыта» нет. Процедура завершена, когда статус поставки = `Поставлено` **и** все её УПД = `Оплачено`.

---

## 6. Производные признаки просрочки

Не отдельные статусы — накладываются поверх:

| Объект | Условие |
|---|---|
| Поставка / процедура | срок по ДД прошёл, а статус поставки ≠ `Поставлено` (или факт позже срока) |
| Оплата (УПД) | срок оплаты прошёл, статус ≠ `Оплачено` → `Просрочена` |

---

## 7. Сводка переходов между блоками

| Действие | Кто | Переход |
|---|---|---|
| Отменить заявку | Комплектация | Заявка: `Ожидает закупки → Отменена` (остаётся в Комплектации) |
| Взять в работу | Закупки | Заявка → блок «В закупке», процедура `Новая` |
| Разбить на торги/процедуры | Закупки | Создаёт процедуры в «В закупке» |
| Выставить рабочий статус | Закупки | Процедура внутри «В закупке» (Приём заявок … На сделку) |
| Отменить процедуру | Закупки | Процедура → `Отменена` (обратимо, остаётся в «В закупке») |
| Передать в сопровождение | Закупки | Процедура: `На сделку` → блок «В сопровождении», статус поставки `Новая` |
| Выставить статус поставки/сделки | Сопровождение | Процедура внутри «В сопровождении» |
| Выставить № УПД | Сопровождение | Создаёт запись в «Оплаты» со статусом `Ожидает оплаты` |
| Провести оплату | Сопровождение | УПД: `Ожидает оплаты → Оплачено` |

---

## 7.1. Видимость: активные vs завершённые/отменённые

На рабочих страницах по умолчанию показываются только **активные** записи. Завершённые и отменённые скрыты под переключателем **«Показать завершённые/отменённые»** (и всегда доступны в отчётах). Счётчики страниц считают активные; показатели дашборда — см. `32-calculations.md`.

| Страница | Активные (по умолчанию) | Под переключателем |
|---|---|---|
| Комплектация («Ожидают закупки») | родители `Ожидает закупки` **без созданных процедур** | `Отменена` |
| В закупке | процедуры `block=zakupka`, статус ≠ `Отменена` | `Отменена` |
| В сопровождении | процедуры `block=soprovozhdenie`, **не завершена** и ≠ `Отменена` | завершённые (`Поставлено` + все УПД `Оплачено`), `Отменена` |
| Оплаты | **все УПД** (полный реестр, как требовалось ранее); переключатель «скрыть оплаченные» — по умолчанию **выключен** | — |

**«Ожидают закупки».** «Взять в работу» создаёт торг + процедуру по умолчанию → у родителя появляются процедуры, и он **покидает** «Ожидают закупки», появляясь в «В закупке». То есть список = родители без процедур.

## 7.2. Предусловия переходов и предупреждения

- Переходы между статусами и блоками **не блокируются обязательными полями** (максимум свободы — поля можно дозаполнять позже).
- **Единственное предупреждение:** УПД без отметки «Серт» попадает в «Требует внимания» (дашборд), но оплату **провести можно** — жёсткой блокировки нет.

---

## 8. Решения и изменения относительно прошлой версии

- 🔄 Перешли на **блочную модель** статусов (свой набор на каждый блок) вместо единого `proc→supp→pay→done`.
- ✅ Статусы блоков «В закупке»/«В сопровождении» — **на уровне процедуры**, независимо; у родителя — только `Ожидает закупки`/`Отменена`.
- 🔄 **Статус поставки — вручную** (ранее было «производный» — решение отменено).
- ✅ **«Оплаты» — параллельный реестр УПД**, не последовательный этап.
- ✅ Отдельного статуса **«Закрыта» нет**; завершённость = `Поставлено` + `Оплачено`.
- ✅ Отменённые заявки/процедуры **остаются в своих блоках** (для отчётности).

---

## 9. Решения

1. ✅ `Отменена` в «В сопровождении» — обратимо, заявка остаётся в блоке (как в закупке).
2. ✅ Рабочие статусы — справочник, выставляются вручную, не обязательно строго по порядку.
3. ✅ `Новая` / `Отменена` — служебные, не входят в справочник (§3).
4. ✅ Активные показываются по умолчанию, завершённые/отменённые — под переключателем (§7.1).
5. ✅ Переходы не блокируются обязательными полями; «Серт» у УПД — предупреждение, не запрет (§7.2).
